# Add a Specialist Worker (Subagent)

A **subagent** is a helper Veda calls in for one specific job, in a fresh context window. The helper sees only the inputs you hand it — not the rest of the conversation. That isolation is the point: it keeps the helper from being biased by what Veda was already thinking.

This is the most advanced extension type, but it is doable as your first contribution if you follow the steps. Read [How Veda thinks](../how-veda-thinks.md) once before starting.

## Should you build a subagent?

A subagent is the right shape when **all** of these are true:

- The job has a clear **input** (what the helper needs) and **output** (what it gives back).
- The helper would do a **better** job if it did **not** see the rest of the chat (e.g., a counter-argument helper that should not see the bull case).
- Veda would call this helper **regularly** — at least once per decision question, not once a year.

If any are false, do not build a subagent. Use one of these instead:

| What you want | Use this |
|---|---|
| A piece of math | [Add a calculator](add-a-script.md) |
| Parse a broker CSV | [Add a broker importer](add-a-broker-importer.md) |
| Add an investor's rules | [Add a framework](add-a-framework.md) |
| A one-time lookup | A small Python script in `scripts/` |

When in doubt, **open an issue first** describing what you want to build. The maintainers will tell you which extension type fits.

## How a subagent looks to Veda

Two things, both files in the repo:

1. **Your contract** at `internal/agents/<name>.md` — a markdown file describing the job, the inputs, the outputs, and what to do when things go wrong. This is the file you write.
2. **A copy** at `.claude/agents/<name>.md` — generated automatically by a script. You do not edit this file. It exists so that AI assistants can discover your subagent.

The contract is just markdown with a few required sections. No coding needed unless your subagent fetches data from somewhere — in that case you also write a short Python script.

## Step-by-step

### Step 1 — Open an issue

Before writing any files, open a GitHub issue. In one short paragraph each, describe:

- **The job.** What does the helper do?
- **The inputs.** What does Veda hand it? (e.g., a ticker symbol, a date range.)
- **The outputs.** What does the helper give back? (e.g., a list of recent news items with source URLs.)
- **Why isolation matters.** Why is this better as a separate helper than as part of Veda's main reasoning?
- **When it runs.** Which of [the 9 stages](../how-veda-thinks.md#the-9-stages) calls it?

A maintainer will respond with feedback. This step prevents you from spending hours building something that will not get merged.

### Step 2 — Copy a template

The cleanest example to copy is `internal/agents/devils-advocate.md`. Open it and read it end to end — the structure is small.

Create a new file `internal/agents/<your-name>.md` and fill in these sections (in this order):

```markdown
---
name: <your-name>
description: <one line — used by AI assistants to discover your subagent>
tools: [Read, WebFetch]   # only what your subagent actually needs
---

# <Your Subagent Name>

## Job

One paragraph. What it does, when Veda calls it, why a separate helper is the right shape.

## Invocation

Which Stage of the pipeline calls your subagent. What triggers it.

## Input contract

The exact shape of what Veda hands your subagent:

```yaml
input:
  ticker: <string, required, e.g. "MSFT">
  archetype: <one of: GROWTH | INCOME_VALUE | TURNAROUND | CYCLICAL>
```

## Output contract

The exact shape of what your subagent gives back:

```yaml
your_subagent:
  status: <ok | partial | failed | skipped>
  result: <the actual content>
  warnings: [<list of strings>]
```

## Behavior in detail

How the helper actually does its job. What sources it consults. What it does when something is missing.

## Failure modes and verification

How does Veda know your subagent did what it said? If your subagent writes files, this section MUST describe how Veda checks the file actually exists and is non-empty before trusting the "ok" status. (This is a common failure on hosts without true subagent isolation.)

## Inline-bridge mode

How Veda runs the same contract on hosts that do not support isolated subagents (ChatGPT, Gemini Gem, Cursor). The contract stays the same; only the isolation is lost.

## Test cases

2 to 5 named examples. For each: an input, the expected output shape, and what should happen.
```

Match the depth of `internal/agents/devils-advocate.md`. Do not pad. Do not over-engineer.

### Step 3 — If your subagent fetches data, write the Python script

Some subagents call a Python script to do their fetching (for example, the existing fundamentals helper calls `scripts/fetch_fundamentals.py`). If yours does, add the script under `scripts/`. The rules:

- It must run on its own from the command line.
- It must print structured JSON to standard output.
- If it fails, it must exit with a non-zero code and include an `error` field in the JSON.

Document the script in [scripts/README.md](../../scripts/README.md) — one row in the table is enough.

If your subagent does not fetch data (it just reasons over the inputs Veda hands it, like the devil's-advocate helper), skip this step entirely.

### Step 4 — Tell Veda when to call your subagent

Open [SKILL.md](../../SKILL.md) and find the Stage your subagent runs in. Add a paragraph that follows this template:

```markdown
**Delegation, where available.** On hosts that support isolated subagents (Claude Code, GitHub Copilot, Antigravity), call the `<your-name>` subagent. Pass it `<list the input fields>`. Use the returned `<output block>` to <describe what Veda does with the answer>.

On hosts without isolated subagents, run the same contract inline.
```

Copy the closest existing pattern in SKILL.md as a starting point. The two patterns shipped today are:

- A reasoning-only subagent (look at how `devils-advocate` is invoked in Stage 7b).
- A file-writing subagent (look at how `fundamentals-fetcher` is invoked in Stage 3).

### Step 5 — Generate the host-facing copy

From the repo root:

```powershell
python scripts/sync_agents.py
```

This reads your `internal/agents/<your-name>.md` and writes a copy to `.claude/agents/<your-name>.md`. The copy carries a banner saying "do not edit". You commit both files.

To check nothing has drifted later:

```powershell
python scripts/sync_agents.py --check
```

It exits with code 0 if everything is in sync. CI runs this on every pull request.

### Step 6 — Submit your pull request

Your PR should include:

- `internal/agents/<your-name>.md` — your contract.
- A new paragraph in [internal/subagents.md](../../internal/subagents.md) (the inventory). Add a row to the status table at the top, and a one-paragraph description in the "Subagent details" section. If your subagent writes files to disk, add a row to the "Subagent write targets" table.
- The new paragraph in `SKILL.md` from Step 4.
- The Python script from Step 3 (if any) and a row in `scripts/README.md`.
- The auto-generated `.claude/agents/<your-name>.md` from Step 5.
- A short worked example in the PR description: one realistic question Veda would call your subagent on, with the expected answer.

Subagents are part of Veda's interface, so the merge bar is high. Expect feedback on the contract shape (the inputs and outputs) before anyone reviews the actual implementation. That is normal.

## Common mistakes a first-time contributor makes

| Mistake | Why it is a problem | What to do instead |
|---|---|---|
| Subagent reads the whole conversation | You lose the isolation that is the entire reason for a subagent | Subagents see only the input contract. If the helper truly needs conversation context, the work belongs inline |
| Subagent writes files but you skip the verification section | On some hosts the helper can claim to have written a file without actually doing so. Veda will trust the lie | Write the verification section. Veda must read the file from disk and check it is non-empty before trusting the "ok" status |
| Output fields are "optional" | Optional fields make it easy for Veda to silently ignore failures | Make every field required. For "no value", use an explicit `null` or empty list |
| You forget the inline-bridge section | Your subagent will not work on ChatGPT, Gemini Gem, or any host without subagent isolation | Always describe how the same contract runs inline |
| You edit the `.claude/agents/` copy directly | Your edit will be overwritten the next time anyone runs `sync_agents.py` | Edit only the canonical file in `internal/agents/` |
| Tool list is "every tool" | Subagents should have the smallest tool surface possible | List only what the helper actually uses (e.g., `Read` and `WebFetch` for a research helper) |
| Same input gives different outputs | Veda cannot be tested or trusted | Tighten the contract and the prompt until the helper is reproducible |
